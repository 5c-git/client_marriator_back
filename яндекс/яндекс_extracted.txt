# Смена API
Яндекс.Смена для потребителя услуг (Провайдера) — это способ быстро закрывать потребность в сменах через единый интеграционный контур: вы создаёте смену, платформа подбирает исполнителя, а вы управляете подтверждением, стартом, фактом и качеством выполнения через события.
## [](ru/#vozmozhnosti)Возможности
- Управление сменами — создание, отмена, приостановка и возобновление поиска исполнителя. Вы задаёте параметры (точка, профессия, тариф, время), а Смена ищет исполнителя.
        
- Работа с исполнителями на сменах — подтверждение или отклонение откликнувшегося исполнителя, старт смены (опционально через QR-код), проставление факта (отработанное время или неявка), оценка выполнения, получение корректировок факта от Смены.
        
- Контроль качества — блокировка/разблокировка исполнителей (глобально или в рамках точки/профессии), добавление в избранные для ускорения повторных выходов (включая режим «только избранные»).
        
- Получение данных исполнителя — ИНН, ФИО, телефон, СНИЛС.

# Жизненный цикл смены
Типовой сценарий от заявки до закрытия:
- Создать смену — Провайдер задаёт точку, профессию, тариф и время.
        
- Получить отклик исполнителя — Смена сообщает, что исполнитель откликнулся.
        
- Принять решение — Провайдер либо подтверждает исполнителя, либо отклоняет (при отклонении поиск продолжается).
        
- Старт смены (опционально с QR-кодом) — Провайдер может выдать код, чтобы у исполнителя появился QR, а затем после проверки зафиксировать старт.
        
- Закрыть смену фактом — Провайдер передаёт факт: отработанное время или неявку.
        
- Оценить качество — Провайдер ставит оценку 1–5.

## [](ru/events#princip-raboty)Принцип работы
Взаимодействие построено на событиях. Все операции — асинхронные.
- Провайдер отправляет события в Смену через `POST /api/v1/events/publish`.
        
- Провайдер получает события от Смены через `GET /api/v1/events/poll`.
        
- Данные исполнителя запрашиваются отдельно через `GET /api/v1/worker/{worker_id}`.
    
    
Типичный сценарий описан в разделе [Жизненный цикл смены](ru/lifecycle).
Важно
Ответ `200 OK` на отправку события означает только приём в очередь. Результат операции (успех или ошибка) приходит позже отдельным событием [`smena.event.result`](ru/smena-event-result) при [поллинге](ru/poll).
## [](ru/events#struktura-sobytiya)Структура события
Каждое событие (в обе стороны) имеет единую структуру:

| Поле | Тип | Описание |
|---|---|---|
| `event_id` | string (UUID v7) | Уникальный идентификатор события |
| `event_type` | string | Тип события (формат: `{источник}.{сущность}.{действие}`) |
| `event_ts` | string (datetime) | Время создания события (UTC) |
| `entity_type` | string | Тип сущности: `shift` или `worker` |
| `entity_id` | string | Идентификатор сущности |
| `payload` | object | Данные события (наполнение зависит от типа) |
Примечание
Все даты и время во всех полях передаются в формате ISO 8601 в UTC (с суффиксом `Z`), например: `2026-01-24T10:00:00Z`.
### [](ru/events#klyuchi-sushnostej)Ключи сущностей
- Смена (`entity_type: shift`): `entity_id` — уникальный идентификатор смены, генерируемый Провайдером. Должен быть уникален для каждой смены и не может быть переиспользован.
        
- Исполнитель (`entity_type: worker`): `entity_id` — идентификатор исполнителя в системе Смены (приходит в событиях от Смены).

## [](ru/events#diagramma-sostoyanij)Диаграмма состояний

| Статус | Описание | Переходы | Без смены состояния |
|---|---|---|---|
| `available` | Смена создана, идёт поиск исполнителя | `smena.shift.signup_worker` → `assigned`; `provider.shift.cancel` → `canceled` | — |
| `assigned` | Исполнитель откликнулся, ожидает подтверждения | `provider.shift.reject_worker` → `available`; `smena.shift.withdraw_worker` → `available`; `provider.shift.cancel` → `canceled`; `provider.shift.start` → `started`; `provider.shift.set_fact` → `finished` / `canceled` | `provider.shift.approve_worker`; `provider.shift.set_code` |
| `started` | Смена начата | `provider.shift.set_fact` → `finished` / `canceled` | — |
| `finished` | Смена завершена | — | `provider.shift.set_fact`; `provider.shift.rate_worker`; `smena.shift.adjust_fact` |
| `canceled` | Смена отменена | `provider.shift.resume` → `available` | — |

# Как отправить событие?
- [Гарантии, которые должен обеспечить Провайдер](ru/publish#garantii,-kotorye-dolzhen-obespechit-provajder)
- [Поведение при ошибках](ru/publish#povedenie-pri-oshibkah)
Endpoint: `POST /api/v1/events/publish`
Тело запроса — объект `Event` (структура описана в разделе [Структура события](ru/events#структура-события)). При успешной обработке метод возвращает `200 OK` без тела ответа.
## [](ru/publish#garantii,-kotorye-dolzhen-obespechit-provajder)Гарантии, которые должен обеспечить Провайдер
- Уникальность `event_id` — повторное событие с тем же `event_id` будет проигнорировано (дедупликация). Метод всё равно вернёт `200 OK`, но событие не будет обработано повторно и дополнительного `smena.event.result` не придёт.
        
- Корректный `event_ts` — для одной и той же сущности (`entity_id` + `entity_type`, см. [Структура события](ru/events#структура-события)) Смена обрабатывает только события с возрастающим `event_ts`. Если событие приходит с `event_ts` меньше, чем у последнего обработанного — оно игнорируется.
    
    
Важно
Пример: если `approve_worker` пришёл с `event_ts = 10:00`, а затем `reject_worker` с `event_ts = 09:59`, то отклонение будет проигнорировано — считается, что исполнитель подтверждён.
## [](ru/publish#povedenie-pri-oshibkah)Поведение при ошибках
Транспорт. Если `publish` вернул ошибку сервера (5xx) или запрос завершился таймаутом, повторите запрос с тем же `event_id`. Благодаря дедупликации событие не будет обработано дважды.
Результат обработки. Ответ `200 OK` на `publish` означает только приём события в очередь. Итог обработки (успех или ошибка) приходит отдельным событием `smena.event.result` при [поллинге](ru/poll). Сопоставляйте его с отправленным событием по `source_event_id`. Подробнее — в разделе [Результат обработки события](ru/smena-event-result).

# Создание смены
- [Payload](ru/provider-shift-create#payload)
- [Пример](ru/provider-shift-create#primer)
`provider.shift.create` — создаёт новую смену. Можно указать избранных исполнителей.
## [](ru/provider-shift-create#payload)Payload

| Поле | Обязательное | Тип | Описание |
|---|---|---|---|
| `site_id` | да | string | ID торговой точки |
| `profession_id` | да | string | ID профессии |
| `payment_id` | да | string | ID тарифа оплаты |
| `start_at` | да | datetime | Дата и время начала (UTC) |
| `length` | да | integer | Длительность смены в минутах (включая перерыв) |
| `rest_length` | да | integer | Длительность перерыва в минутах |
| `favorites_worker_ids` | нет | string[] | ID избранных исполнителей |
| `favorites_only` | нет | boolean | Если `true` — смена будет отменена, если ни один из избранных исполнителей не сможет выйти |
## [](ru/provider-shift-create#primer)Пример
`curl -X POST "$HOST/api/v1/events/publish" \
  -H "Content-Type: application/json" \
  -d '{
    "event_id": "550e8400-e29b-71d4-a716-446655440001",
    "event_type": "provider.shift.create",
    "event_ts": "2026-01-23T08:00:00Z",
    "entity_type": "shift",
    "entity_id": "shift-001",
    "payload": {
      "site_id": "site-1",
      "profession_id": "prof-1",
      "payment_id": "pay-1",
      "start_at": "2026-01-24T10:00:00Z",
      "length": 480,
      "rest_length": 60,
      "favorites_worker_ids": ["worker-42"],
      "favorites_only": false
    }
  }'
`

        

            
        
    
    
Важно
`favorites_only = true`: если все избранные исполнители недоступны, смена автоматически переходит в статус `canceled`. На каждого недоступного исполнителя приходит событие `smena.shift.withdraw_worker` с причиной (`favorite_not_found`, `favorite_not_confirmed`, `favorite_has_intersection`).

Провайдер передаёт код, на основе которого в приложении исполнителя отображается QR. Исполнитель показывает QR на точке, Провайдер сканирует его и проверяет по своей логике. После успешной проверки Провайдер отправляет событие `provider.shift.start`, фиксируя начало смены.
Код можно обновлять повторной отправкой события. Если указан `expires_at`, после истечения срока код перестаёт отображаться в приложении.

| `expires_at` | нет | datetime | Срок действия кода (UTC). После истечения код перестаёт отображаться |

# Блокировка исполнителя
- [Payload](ru/provider-worker-block#payload)
- [Пример](ru/provider-worker-block#primer)
`provider.worker.block` — блокирует исполнителя. Область блокировки определяется переданными полями:
- Указаны `site_id` и/или `profession_id` — блокировка на конкретной точке/профессии.
        
- Не указаны — глобальная блокировка.
## [](ru/provider-worker-block#payload)Payload

| Поле | Обязательное | Тип | Описание |
|---|---|---|---|
| `reason` | да | string | Причина блокировки |
| `block_type` | да | string | `temporary` — временная, `permanent` — постоянная |
| `blocked_until` | да (для `temporary`) | datetime | До какого момента действует блокировка (UTC). Обязателен при `block_type = temporary` |
| `site_id` | нет | string | ID торговой точки |
| `profession_id` | нет | string | ID профессии |
| `comment` | нет | string | Комментарий |
Допустимые значения `reason`:

| Значение | Описание |
|---|---|
| `incident` | Инцидент |
| `invalid_documents` | Недействительные документы |
| `intoxicated` | Алкогольное/наркотическое опьянение |
| `spoof_account` | Поддельный аккаунт |
| `no_mandatory_training` | Не пройдено обязательное обучение |
| `low_performance` | Низкая эффективность |
| `low_qualification` | Низкая квалификация |
| `other` | Другое |

# Как получить событие?
- [Параметры запроса](ru/poll#parametry-zaprosa)

- [Рекомендации](ru/poll#rekomendacii)

Endpoint: `GET /api/v1/events/poll`
## [](ru/poll#parametry-zaprosa)Параметры запроса

| Параметр | Обязательный | Тип | Описание |
|---|---|---|---|
| `last_event_id` | нет | string (UUID v7) | ID последнего полученного события (курсор) |
| `limit` | нет | integer | Количество событий в ответе (1–1000, по умолчанию 100) |
| `event_type` | нет | string[] | Фильтр по типам событий. Если не указан — все типы |
`curl "$HOST/api/v1/events/poll?last_event_id=550e8400-e29b-71d4-a716-446655440000&limit=100"
`

        

            
        
    
    
С фильтром по типу:
`curl "$HOST/api/v1/events/poll?last_event_id=550e8400-e29b-71d4-a716-446655440000&limit=100&event_type=smena.shift.signup_worker&event_type=smena.shift.withdraw_worker"
`
## [](ru/poll#format-otveta)Формат ответа
`{
  "events": [ ... ],
  "has_next": true
}
`

        

            
        
    
    
        
- `events` — массив событий.
        
- `has_next` — `true`, если после текущего батча есть ещё события.
    
    
Пример ответа:
`{
  "events": [
    {
      "event_id": "019468a0-7741-7000-a278-afb18b4e244f",
      "event_type": "smena.shift.signup_worker",
      "event_ts": "2026-01-24T09:00:00Z",
      "entity_type": "shift",
      "entity_id": "shift-001",
      "payload": {
        "worker_id": "worker-42",
        "is_favorite": false
      }
    },
    {
      "event_id": "019468a0-7741-7001-b389-cfc29c5f355a",
      "event_type": "smena.shift.withdraw_worker",
      "event_ts": "2026-01-24T09:05:00Z",
      "entity_type": "shift",
      "entity_id": "shift-002",
      "payload": {
        "worker_id": "worker-15",
        "is_favorite": true,
        "reason": "cancelled_by_worker"
      }
    }
  ],
  "has_next": false
}
`
## [](ru/poll#rekomendacii)Рекомендации
- Для получения событий в режиме, близком к реальному времени, рекомендуется опрашивать метод раз в секунду. Частота может быть снижена, если оперативность не критична.
        
- Используйте `event_id` последнего полученного события как `last_event_id` для следующего запроса.
        
- Доступны события не старше 24 часов.
## [](ru/poll#garantii-smeny)Гарантии Смены
- `event_id` уникален и монотонно возрастает — используйте его как курсор.
        
- События возвращаются в порядке, в котором они должны быть обработаны.

`smena.event.result` — Смена создаёт это событие для каждого обработанного provider-события: при успехе и при ошибке. Провайдер получает его при [поллинге](ru/poll) вместе с остальными событиями.
`entity_type` и `entity_id` совпадают с сущностью из исходного provider-события (смена или исполнитель).
## [](ru/smena-event-result#payload)Payload

| Поле | Обязательное | Тип | Описание |
|---|---|---|---|
| `source_event_id` | да | string (UUID) | `event_id` отправленного provider-события |
| `source_event_type` | да | string | `event_type` отправленного события (один из `provider.*`) |
| `status` | да | string | `success` — обработано успешно; `error` — ошибка |
| `error_code` | при `status=error` | string | Код ошибки (см. каталог ниже) |
| `error_message` | при `status=error` | string | Описание ошибки (для логирования) |
## [](ru/smena-event-result#primer-pri-uspehe)Пример при успехе
`{
  "event_id": "019468a0-7741-7002-c490-dde30d6f466b",
  "event_type": "smena.event.result",
  "event_ts": "2026-01-23T10:01:00Z",
  "entity_type": "shift",
  "entity_id": "shift-001",
  "payload": {
    "source_event_id": "550e8400-e29b-71d4-a716-446655440001",
    "source_event_type": "provider.shift.cancel",
    "status": "success"
  }
}
`
## [](ru/smena-event-result#primer-pri-oshibke)Пример при ошибке
`{
  "event_id": "019468a0-7741-7003-d5e41e7f577c",
  "event_type": "smena.event.result",
  "event_ts": "2026-01-23T10:01:00Z",
  "entity_type": "shift",
  "entity_id": "shift-001",
  "payload": {
    "source_event_id": "550e8400-e29b-71d4-a716-446655440002",
    "source_event_type": "provider.shift.create",
    "status": "error",
    "error_code": "invalid_site_id",
    "error_message": "Site site-999 not found"
  }
}
`
## [](ru/smena-event-result#katalog-kodov-oshibok)Каталог кодов ошибок
При `status=error` в payload присутствуют `error_code` и `error_message`.
### [](ru/smena-event-result#obshie-—-dlya-lyubogo-provider-sobytiya)Общие — для любого provider-события

| Код | Описание | Рекомендуемое действие |
|---|---|---|
| `entity_not_found` | Сущность не найдена (смена не создана, исполнитель неизвестен) | Проверить `entity_id` и порядок событий |
| `invalid_state_transition` | Операция невозможна в текущем состоянии | Проверить состояние сущности и отправить корректное событие |
| `event_outdated` | Событие проигнорировано: `event_ts` меньше последнего обработанного для этой сущности | Обеспечить монотонное возрастание `event_ts` |
| `internal_error` | Внутренняя ошибка Смены | Повторить операцию с новым `event_id`. При повторении — в поддержку |
### [](ru/smena-event-result#)Создание смены (`provider.shift.create`)

| Код | Описание | Рекомендуемое действие |
|---|---|---|
| `invalid_site_id` | Торговая точка не найдена | Проверить `site_id` |
| `invalid_profession_id` | Профессия не найдена | Проверить `profession_id` |
| `invalid_payment_id` | Тариф оплаты не найден | Проверить `payment_id` |
### [](ru/smena-event-result#operacii-s-ispolnitelem-na-smene)Операции с исполнителем на смене
Применимо к: `provider.shift.approve_worker`, `provider.shift.reject_worker`, `provider.shift.set_code`, `provider.shift.start`, `provider.shift.set_fact`.

| Код | Описание | Рекомендуемое действие |
|---|---|---|
| `worker_not_assigned` | На смене нет указанного исполнителя (или другой назначен) | Дождаться `smena.shift.signup_worker` перед операцией |
| `worker_already_withdrawn` | Исполнитель уже снят со смены | Дождаться нового отклика |
## [](ru/smena-event-result#rekomendacii)Рекомендации
- Корреляция: храните отправленные события по `event_id` и при получении `smena.event.result` сопоставляйте по `source_event_id`.
        
- Успех: при `status=success` считайте операцию выполненной; можно обновить локальное состояние.
        
- Ошибка: логируйте `error_code` и `error_message`. Не повторяйте то же событие с тем же `event_id` — для повторной попытки создайте новое событие с новым `event_id` и актуальным `event_ts`.
        
- Неизвестные коды: обрабатывайте как общую ошибку (логировать, уведомлять). Новые коды могут добавляться без смены версии API.
    
    
Подробнее о поведении при ошибках — в разделе [Отправка событий](ru/publish).

# Авторизация
Доступ к API осуществляется по API-ключу в связке с IP whitelist. Ключ выдаётся при подключении и передаётся в заголовке каждого запроса:
`Authorization: Bearer <token>
`

        

            
        
    
    
При невалидном или отсутствующем ключе метод возвращает `401 Unauthorized`.
Примечание
Во всех примерах запросов в документации заголовок авторизации опущен для краткости.

# Ошибки валидации
Тело запроса валидируется по схеме. Если запрос не соответствует схеме, метод вернёт ошибку:
`{
    "code": "openapi_validation_request_error",
    "message": "Invalid request",
    "details": {
        "error": "request body has an error: doesn't match the schema: Error at \"/message\": Field must be set to string or not be present"
    }
}
`

        

            
        
    
    
        
- `code` — всегда `openapi_validation_request_error`.
        
- `message` — краткое описание.
        
- `details.error` — подробности о том, какое поле не прошло валидацию.

# Ограничения

| Метод | Лимит |
|---|---|
| `POST /api/v1/events/publish` | 40 запросов/сек |
| `GET /api/v1/events/poll` | 60 запросов/мин |
| `GET /api/v1/worker/{worker_id}` | 100 запросов/мин |
Максимальный `limit` при поллинге — 1000 событий за запрос.
При превышении лимита метод возвращает `429 Too Many Requests` с заголовком `Retry-After`, указывающим через сколько секунд можно повторить запрос.

# Обратная совместимость
API развивается с сохранением обратной совместимости. При интеграции учитывайте следующее:
- Новые поля могут быть добавлены в любой `payload` — игнорируйте неизвестные поля.
        
- Новые типы событий (`event_type`), причины (`reason`), коды ошибок (`error_code`) и другие строковые значения могут появляться без предупреждения — корректно обрабатывайте неизвестные значения (логируйте и пропускайте).
        
- Основная структура события (кроме `payload`) не изменяется без повышения версии API.

## [](ru/faq#)Откуда берутся `site_id`, `profession_id`, `payment_id`?
Идентификаторы торговых точек, профессий и тарифов создаются на стороне Смены и передаются Провайдеру при подключении. В первой версии API нет интерфейса для их самостоятельного создания или изменения.
## [](ru/faq#1)Откуда взять `worker_id` для `favorites_worker_ids`?
`worker_id` появляется в событиях [`smena.shift.signup_worker`](ru/smena-shift-signup-worker). Чтобы использовать исполнителя как избранного в будущем, сохраняйте `worker_id` из предыдущих откликов. Поле `favorites_worker_ids` описано в разделе [Создание смены](ru/provider-shift-create).
## [](ru/faq#kak-redaktirovat-smenu)Как редактировать смену?
Отдельного события для редактирования нет. Отмените смену (`provider.shift.cancel`) и создайте новую (`provider.shift.create`). Новая смена должна иметь новый `entity_id` — переиспользовать идентификатор отменённой смены нельзя.
## [](ru/faq#,-esli-vse-izbrannye-ispolniteli-nedostupny)Что произойдёт при `favorites_only = true`, если все избранные исполнители недоступны?
На каждого избранного исполнителя придёт `smena.shift.withdraw_worker` с соответствующей причиной (`favorite_not_found`, `favorite_not_confirmed`, `favorite_has_intersection`). После исчерпания всех — смена будет автоматически отменена.
## [](ru/faq#2)Что делать, когда `has_next = true`?
Немедленно запросите следующий батч, передав `event_id` последнего полученного события как `last_event_id`. Повторяйте до `has_next = false`, затем вернитесь к обычному поллингу раз в секунду. Подробнее — в разделе [Получение событий](ru/poll).
## [](ru/faq#chto-esli-endpoint-ne-oprashivalsya-bolee-24-chasov)Что если endpoint не опрашивался более 24 часов?
События старше 24 часов недоступны. Для восстановления состояния обратитесь в поддержку Смены.
## [](ru/faq#nuzhna-li-deduplikaciya-na-storone-provajdera-pri-pollinge)Нужна ли дедупликация на стороне Провайдера при поллинге?
Нет. При корректном использовании `last_event_id` каждый запрос возвращает только новые события. Если по какой-то причине вы получили одно и то же событие дважды — просто обработайте его повторно или пропустите по `event_id`.
